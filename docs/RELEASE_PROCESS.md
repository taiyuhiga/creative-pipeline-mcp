## Assets

Generate release assets:

```bash
npm run release:assets
```

This writes:

```text
dist/release/
  creative-pipeline-mcp-<version>.tgz
  creative-pipeline-mcp-example-artifacts-<version>.zip
  checksums.txt
dist/premiere-cep/
  creative-pipeline-mcp-premiere-cep-panel-<version>.zip
  premiere-cep-checksums.txt
dist/zxp/
  creative-pipeline-mcp-premiere-cep.zxp
  zxp-checksums.txt
```

Upload:

```bash
gh release upload v<version> dist/release/* --clobber
gh release upload v<version> dist/premiere-cep/* --clobber
gh release upload v<version> dist/zxp/* --clobber
```

## npm Publish Readiness

Check name and package metadata:

```bash
npm view creative-pipeline-mcp version
npm pack --dry-run
npm run smoke:npm-install
npm publish --dry-run --provenance
```

Check dist-tags before and after publishing:

```bash
npm view creative-pipeline-mcp dist-tags --json
```

Pre-release versions must use `alpha`. Stable `1.0.0` must be the first intentional release published with `latest`. Users who want the newest alpha must install:

```bash
npm install creative-pipeline-mcp@alpha
```

Do not tell users to install the unqualified `creative-pipeline-mcp` package until v1 stable is intentionally published.

## npm Dist-Tag Maintenance

Expected pre-v1 state:

```text
alpha  -> newest 0.x alpha
latest -> unset, or an older accidental alpha that must not be promoted as stable
```

If `latest` points to an alpha package before v1, clean it up only from an authenticated npm account with publish rights:

```bash
npm whoami
npm dist-tag rm creative-pipeline-mcp latest
npm view creative-pipeline-mcp dist-tags --json
```

If `npm whoami` returns `E401`, reauthenticate with `npm login` before attempting tag maintenance. If npm asks for a second factor, use a current OTP from the account's configured 2FA method. Do not use recovery codes unless the account owner explicitly instructs you to do so for that command. If authentication is unavailable, keep the stable release issue blocked and document that users must use `@alpha`.

Local publishing requires an authenticated npm session and a second factor. Do not pass `--provenance` from a local shell; npm provenance is generated by trusted publishing in CI. Alpha releases should use the `alpha` dist-tag:

```bash
npm publish --access public --tag alpha --otp <one-time-code-or-recovery-code>
```

GitHub Actions publishing uses npm trusted publishing. Configure npmjs.com with:

```text
Owner: taiyuhiga
Repository: creative-pipeline-mcp
Workflow: npm-publish.yml
Environment: leave empty unless a GitHub environment is added
```

Then set the repository variable `NPM_TRUSTED_PUBLISHING_ENABLED=true`. The workflow publishes from release tags only after tests, adapter checks, Premiere project-delivery example, CEP package verification, npm pack dry-run, and package-version/tag matching pass. Pre-release package versions publish with the `alpha` dist-tag; stable versions publish with `latest`.
